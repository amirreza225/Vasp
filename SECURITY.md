# Security Policy

## Supported Versions

Only the latest published version of each package receives security fixes.

| Package | Supported |
|---------|-----------|
| `vasp-cli` (latest) | Yes |
| `@vasp-framework/core` (latest) | Yes |
| `@vasp-framework/parser` (latest) | Yes |
| `@vasp-framework/generator` (latest) | Yes |
| `@vasp-framework/runtime` (latest) | Yes |
| Older versions | No |

## Reporting a Vulnerability

**Do not open a public GitHub issue for security vulnerabilities.**

Please report them privately via [GitHub Security Advisories](https://github.com/amirreza225/Vasp/security/advisories/new). This keeps the details confidential until a fix is released.

Include as much of the following as possible:

- A description of the vulnerability and its potential impact
- Steps to reproduce or a minimal proof of concept
- The affected package(s) and version(s)
- Any suggested mitigation or fix

You will receive a response within **72 hours** acknowledging the report. If the vulnerability is confirmed, a fix will be prioritized and a new version published. You will be credited in the release notes unless you prefer to remain anonymous.

## Scope

This policy covers vulnerabilities in the Vasp framework packages themselves. It does **not** cover:

- Applications generated by Vasp (security of generated output is the responsibility of the developer using Vasp)
- Third-party dependencies (report those to the respective upstream projects)
- Issues requiring physical access to a machine

## Security Considerations for Generated Apps

Vasp generates production-ready applications. The following security features are included by default in generated output:

- **Password hashing** — Argon2id via `Bun.password.hash()`, never plain-text or weak hashes
- **JWT authentication** — signed with `jose`, verified on every protected request
- **Rate limiting** — IP-based sliding-window limiter on every generated server
- **Input validation** — entity field validators enforced at the API layer
- **CORS** — configurable per app, defaults to same-origin
- **Environment variable validation** — startup-time checks for required env vars

If you believe a generated application has a structural security flaw (e.g. a generator produces insecure code by default), that qualifies as a vulnerability in Vasp and should be reported here.
